## v1 Requirements — ALL COMPLETE ✅

### Crawling (CRAWL)

- [x] **CRAWL-01**: User can submit a URL and receive an analysis result within 60 seconds via async job queue
  - _Outcome: Validated. QStash-backed async queue; job lifecycle pending→crawling→extracting→analyzing→complete; Phase 1 smoke test confirmed 202+jobId response._
- [x] **CRAWL-02**: Crawler executes JavaScript and waits for SPA hydration before signal extraction (handles React, Next.js, Vue)
  - _Outcome: Validated. `waitForSpaHydration()` detects __NEXT_DATA__, __vue_app__, __nuxt, ReactFiber keys. Verified on nextjs.org (02-06 smoke test)._
- [x] **CRAWL-03**: Crawler runs each analysis in both mobile (375px, throttled) and desktop (1440px) viewport profiles
  - _Outcome: Validated. `runDualViewportCrawl` runs mobile 375×812 (slow-3G throttle) + desktop 1440×900. Confirmed in crawler logs (02-06)._

### Signal Extraction (SIG)

- [x] **SIG-01**: System extracts DOM/HTML signals — layout depth, element counts, ARIA attributes, semantic markup quality, form structure, CTA visibility
  - _Outcome: Validated. `extractors/dom.ts` produces `DOMSignals`; consumed by Stage 1 scorer DOM_RULES._
- [x] **SIG-02**: System extracts CSS signals — animation performance, layout complexity, paint-triggering properties, unused CSS (via Playwright coverage API), font loading patterns
  - _Outcome: Validated. `extractors/css.ts` produces `CSSSignals`; consumed by Stage 1 CSS_RULES._
- [x] **SIG-03**: System extracts JavaScript signals — bundle transfer size, chunk count, render-blocking vs async/deferred classification, framework fingerprint (loading behavior only; no code-quality analysis of minified bundles)
  - _Outcome: Validated. `extractors/js.ts` produces `JSSignals`; consumed by Stage 1 JS_RULES._
- [x] **SIG-04**: System extracts network/asset signals — full request timing waterfall (DNS/TLS/TTFB/download), render-blocking asset identification, CDN usage, image compression and sizing
  - _Outcome: Validated. `extractors/network.ts` + HAR recording produces `NetworkSignals`; consumed by Stage 1 NETWORK_RULES._

### AI Pipeline (AI)

- [x] **AI-01**: Rule-based scoring stage classifies each extracted signal into Critical / High / Medium / Low severity using deterministic thresholds (no LLM at this stage)
  - _Outcome: Validated. `scoreSignals()` in `stage1-scorer.ts` — 23 threshold rules, pure synchronous function. Smoke test confirmed severity values in {1,2,3,4}._
- [x] **AI-02**: LLM reasoning stage explains why each scored issue matters and generates causality edges between issues — grounded only in signals that passed scoring; LLM is not permitted to invent issues
  - _Outcome: Validated. `runStage2Reasoning()` via Groq llama-3.3-70b-versatile with zod-validated output; mechanism field non-nullable from PERMITTED_MECHANISMS list (13 strings). 5 CausalEdges confirmed in 03-05 smoke test._
- [x] **AI-03**: LLM narration stage generates a 2–4 paragraph plain-English summary readable by non-engineers (PMs, UX leads, agency clients)
  - _Outcome: Validated. `runStage3Narration()` via Groq; `NarrativeResult` with summary, perceivedPerformance, technicalPerformance, recommendations. Non-empty fields confirmed in 03-05 smoke test._
- [x] **AI-04**: AI pipeline explicitly distinguishes perceived performance (how slow it feels) from technical performance (what the metrics say) in both the issue list and narrative
  - _Outcome: Validated. System prompt explicitly requires [PERCEIVED PERFORMANCE] and [TECHNICAL PERFORMANCE] section markers. NarrativeSection renders them in separate sections._

### Dashboard & Output (DASH)

- [x] **DASH-01**: Dashboard displays a ranked issue list ordered by UX impact severity (Critical first), with per-issue plain-English explanation and the signal evidence that triggered it
  - _Outcome: Validated. `result.issues` ordered `severity: 'desc'` in Prisma query; IssueCard renders per-issue with SeverityBadge. 117 tests GREEN._
- [x] **DASH-02**: Dashboard displays the plain-English narrative summary prominently as the primary output section
  - _Outcome: Validated. NarrativeSection rendered before issue list in page.tsx; displays all 4 sub-sections. 117 tests GREEN._
- [x] **DASH-03**: Dashboard renders a causality graph (directed graph) showing technical cause → perceived effect chains using React Flow; only displayed when chain quality meets credibility threshold
  - _Outcome: Validated. `meetsCredibilityThreshold()` gates rendering; CausalityGraph (React Flow with Background/Controls/MiniMap) renders when met; GraphAbsent shown otherwise. 117 tests GREEN._
- [x] **DASH-04**: User can share analysis results via a persistent link or copy the full output to clipboard without requiring an account
  - _Outcome: Validated. /results/[jobId] is a public route (no auth); ShareButton copies window.location.href. 117 tests GREEN._

### Infrastructure & Security (INFRA)

- [x] **INFRA-01**: URL submission endpoint validates and blocks SSRF targets — private IP ranges (RFC-1918), localhost, link-local addresses, and cloud metadata endpoints (169.254.x.x)
  - _Outcome: Validated. `src/lib/ssrf.ts` validates before DB write in /api/analyze; SSRF Layer 2 in `browser.ts` Playwright context routes. Phase 1 smoke test confirmed blocking of 192.168.1.1._
- [x] **INFRA-02**: URL submission is rate-limited per IP to prevent abuse and uncontrolled Claude API spend
  - _Outcome: Validated. Vercel Edge Middleware with Upstash sliding window: 5/hour + 20/day per IP. Phase 1 smoke test confirmed 429 on 6th request._
- [x] **INFRA-03**: Final analysis results (scored issues, narrative, causality edges) are persisted to PostgreSQL; raw signal payloads are ephemeral and not stored
  - _Outcome: Validated. processor.ts: `const signals = { mobile, desktop }` never written to DB. Only Result (narrative JSON) + Issue[] + CausalEdge[] persisted. Confirmed in 01-08 smoke test (no raw signal columns in DB)._
- [x] **INFRA-04**: Frontend polls a job-status endpoint during analysis so the user sees meaningful progress state (queued → crawling → extracting → analyzing → complete)
  - _Outcome: Validated. JobStatusBadge polls GET /api/jobs/[jobId] every 2s via setInterval; auto-redirects to /results/[jobId] on complete. Phase 1 smoke test confirmed status badge transitions._

## Out of Scope

Explicitly excluded from FeelTrace v1 and v2. Adding these would change the product category.

- **Runtime tracking / session replays** — not replacing LogRocket, FullStory, or Hotjar; no snippet injection; no live user data collection
- **Real user monitoring (RUM)** — no event collection from actual user sessions
- **Raw Lighthouse-style composite scores (0–100)** — Lighthouse does this better; referencing them in narrative context only
- **Full axe violation checklists** — axe does this better; surfacing accessibility issues only when they contribute to a friction chain
- **Code-level fix suggestions (line-by-line)** — advisory only, not code review
- **Browser extension** — contradicts zero-install value proposition
- **Custom rule configuration / scoring weights** — opinionated defaults are a feature; enterprise customization is v3+
- **Performance benchmarking / competitor comparison** — requires benchmark dataset not available in v1
- **User accounts in MVP** — single-use, shareable-link model only
