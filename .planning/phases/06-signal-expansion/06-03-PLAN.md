# Plan 06-03: Pipeline Verification & Integration Test

**Phase:** 06 — Signal Expansion
**Plan:** 03 of 3
**Requirements:** SIGNAL-01, SIGNAL-02, SIGNAL-03
**Goal:** Verify that all three signal types flow end-to-end from the scorer through `runAIPipeline`, confirm Stage 2 prompt handles the new `signal_source` prefixes without regression, and run the full test suite and typecheck to close Phase 6.

---

## Context

Plans 06-01 and 06-02 implement the three signal types independently. This plan adds the integration-level test that exercises all three together: mock PSI response + mock axe violations → `scoreExternalSignals` + `scoreAxeViolations` → combined `scoredIssues` array → Stage 2 input. It also audits the Stage 2 system prompt to ensure `cwv.`, `lighthouse.`, and `axe.` prefixed `signal_source` values do not confuse the LLM instruction text, and then runs the full `npm test` + `npm run typecheck` to confirm no regressions.

**Prerequisite:** Plans 06-01 and 06-02 must be complete.

---

## Must-Haves

1. An integration test at `crawler/src/pipeline/stage1-integration.test.ts` exercises the full Stage 1 path (existing `scoreSignals` + `scoreExternalSignals` + `scoreAxeViolations`) with mocked inputs and asserts the combined `ScoredIssue[]` contains expected entries from all three signal types.
2. `crawler/src/pipeline/stage2-reasoner.ts` (read-only audit) does not hard-code signal_source prefix allowlists that would reject `cwv.*`, `lighthouse.*`, or `axe.*` sources. If any such allowlist is found, it must be extended or removed.
3. `npm run typecheck` (runs `tsc --noEmit` over the `crawler/` package) exits with code 0.
4. `npm test` (Vitest) exits with code 0 with all tests passing, including the new unit tests from Plans 06-01 and 06-02.
5. The `_signals` sub-key written to `Result.tech_stack` is a plain-serialisable object (no `Buffer`, no circular refs) — confirmed by the integration test serializing it through `JSON.stringify` without throwing.

## Tasks

### Wave 1: Stage 2 Prompt Audit

- [ ] **Read `crawler/src/pipeline/stage2-reasoner.ts`** and check for any hard-coded signal_source prefix allowlists, enum strings, or prompt instructions that reference specific `signal_source` prefixes (e.g., `networkSignals`, `jsSignals`). If such lists exist:

  - If no such restriction exists (most likely), no change is needed — record "no signal_source allowlist found" in the integration test file as a comment.

### Wave 2: Integration Test

- [ ] **Create `crawler/src/pipeline/stage1-integration.test.ts`** — exercises Plans 06-01 and 06-02 together:
  - Import `scoreSignals` from `./stage1-scorer`, `scoreExternalSignals` and `scoreAxeViolations` from `./stage1-external-scorer`.
  - Build a minimal `CrawlPass` stub (mobile + desktop) with all required fields set to zero/empty values (avoids triggering any existing threshold rules). The stub must satisfy the full `CrawlPass` TypeScript shape.
  - Build a mock `ExternalSignals` with:
    - `cwv: { lcp_ms: 5000, cls_raw: 30, inp_ms: 600, origin_fallback: false }` — all three metrics above Critical thresholds.
    - `lighthouse: { performance: 0.4, accessibility: 0.75, seo: 0.6, bestPractices: 0.9 }` — performance Critical, a11y High, seo Medium.
  - Build mock `AxeViolation[]` with 3 violations: one `critical`, one `serious`, one `moderate`.
  - Combine: `const allIssues = [...scoreSignals(mobile, desktop), ...scoreExternalSignals(externalSignals), ...scoreAxeViolations(axeViolations)]`.
  - Assert:
    - `allIssues` contains at least one issue with `signal_source` starting with `'cwv.'`.
    - `allIssues` contains at least one issue with `signal_source` starting with `'lighthouse.'`.
    - `allIssues` contains at least one issue with `signal_source` starting with `'axe.'`.
    - The CWV LCP Critical issue has `severity: 4`.
    - The axe critical violation has `severity: 4`.
    - `JSON.stringify({ cwv: externalSignals.cwv, lighthouse: externalSignals.lighthouse })` does not throw (serialisability check).
  - No network calls, no Prisma, no LLM — pure in-memory unit test.

## Notes

- **Minimal CrawlPass stub:** The integration test's `CrawlPass` stub must include every non-optional field. Use `as CrawlPass` only if the stub is genuinely full-shaped — do not mask missing fields with a type assertion if they would cause a runtime error in `scoreSignals`. Set numeric fields to `0`, arrays to `[]`, strings to `''`, and booleans to `false`.
- **Stage 2 prompt audit is read-only by default:** The audit step is investigative. Most likely `stage2-reasoner.ts` uses free-text LLM instructions referencing `signal_source` by example, not by allowlist. In that case, no code change is needed — the LLM will handle new prefixes generically.
- **`tsc` in crawler is standalone:** The `crawler/` package has its own `tsconfig.json` and its own `tsc` invocation. Run `npm run typecheck` from inside `crawler/`, not from the monorepo root.
- **Vitest config:** Check that `crawler/` has a Vitest config (or `vitest` field in `package.json`) that covers `src/**/*.test.ts`. If the test runner is not yet configured in the crawler package, add a minimal `vitest.config.ts` with `{ test: { include: ['src/**/*.test.ts'] } }` — but only if tests cannot be discovered otherwise.
- **TypeScript tuple inference through `Promise.race`:** If TypeScript cannot resolve the destructured return type from the new `Promise.race([Promise.all([...]), ...])` pattern in `processor.ts`, add an explicit type annotation:
  ```
  const [[crawlResult, psiResult]] = await Promise.race<
    [Awaited<ReturnType<typeof runDualViewportCrawl>>, ExternalSignals | null][]
  >([...])
  ```
  Or use a two-step approach: define the `Promise.all` result type separately then pass it to `Promise.race`.
