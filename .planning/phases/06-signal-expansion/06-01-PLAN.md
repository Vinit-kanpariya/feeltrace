# Plan 06-01: PSI Integration (CWV + Lighthouse)

**Phase:** 06 — Signal Expansion
**Plan:** 01 of 3
**Requirements:** SIGNAL-01, SIGNAL-03
**Goal:** Fetch CrUX field data (LCP, CLS, INP) and Lighthouse category scores in parallel with the Playwright crawl via the PageSpeed Insights API, store raw values in `Result.tech_stack._signals`, and emit scored issues through a new `stage1-external-scorer.ts`.

---

## Context

This plan wires SIGNAL-01 (Core Web Vitals via CrUX) and SIGNAL-03 (Lighthouse scores) into the crawler pipeline. A single PSI call fetches both signal types simultaneously, runs in parallel with `runDualViewportCrawl` via `Promise.all` wrapped in the existing 55-second SLA race, and the results travel through a new `ExternalSignals` struct into a new scorer file. The existing `stage1-scorer.ts` and `scoreSignals()` are left completely unchanged. `PAGESPEED_API_KEY` is soft-required: if absent the service warns at startup and skips PSI gracefully.

---

## Must-Haves

1. `crawler/src/lib/types.ts` exports `AxeViolationNode`, `AxeViolation`, `CWVMetrics`, `LighthouseScores`, and `ExternalSignals` interfaces exactly matching the shapes in the RESEARCH.md type contracts. `CrawlPass` has `axeViolations?: AxeViolation[]` added.
2. `crawler/src/lib/psi.ts` exports `fetchPSISignals(url: string, timeoutMs: number): Promise<ExternalSignals | null>` — returns `null` (not throws) on timeout, missing API key, or non-200 HTTP status.
3. `crawler/src/index.ts` logs `[feeltrace-crawler] PAGESPEED_API_KEY not set — PSI/CWV/Lighthouse signals disabled` as a `console.warn` (not `console.error`, never `process.exit(1)`) when the key is absent.
4. `crawler/src/pipeline/stage1-external-scorer.ts` exports `scoreExternalSignals(signals: ExternalSignals): ScoredIssue[]` and `scoreAxeViolations(violations: AxeViolation[]): ScoredIssue[]` with all CWV and Lighthouse threshold rules from the research notes.
5. `crawler/src/processor.ts` runs `runDualViewportCrawl` and `fetchPSISignals` in parallel via `Promise.all` inside the existing `Promise.race` SLA guard; the `psiResult` flows into `runAIPipeline` as an additional argument.
6. `crawler/src/pipeline/run-pipeline.ts` accepts `externalSignals: ExternalSignals | null` as a fourth parameter, concatenates `scoreExternalSignals(externalSignals)` results onto `scoredIssues`, and stores raw CWV + Lighthouse values in `Result.tech_stack` under a `_signals` sub-key.
7. Unit tests in `crawler/src/lib/psi.test.ts` verify: (a) returns `null` when `PAGESPEED_API_KEY` is absent, (b) returns `null` on timeout (mock `fetch` to hang), (c) correctly parses a stubbed PSI response for both CWV and Lighthouse fields including the `best-practices` bracket-notation key.
8. Unit tests in `crawler/src/pipeline/stage1-external-scorer.test.ts` verify all CWV threshold rules (LCP Critical >4000, LCP High >2500, CLS Critical raw >25, CLS High raw >10, INP Critical >500, INP High >200) and all Lighthouse threshold rules (perf <0.5 Critical, perf <0.7 High, a11y <0.8 High, a11y <0.9 Medium, seo <0.7 Medium).

### Wave 1: PSI Client, Startup Warning, Scorer

- [ ] **Create `crawler/src/lib/psi.ts`** — implement `fetchPSISignals`:
  - Build the PSI URL: `https://www.googleapis.com/pagespeedonline/v5/runPagespeed?url=<encoded>&strategy=MOBILE&category=PERFORMANCE&category=ACCESSIBILITY&category=SEO&category=BEST_PRACTICES&key=<PAGESPEED_API_KEY>`.
  - If `PAGESPEED_API_KEY` is not set, immediately return `null`.
  - Wrap the fetch in `AbortController` + `setTimeout(timeoutMs)` — abort and return `null` on timeout.
  - On non-200 response, log `[psi] HTTP <status> for <url> — skipping` and return `null`.
  - Parse `loadingExperience.metrics` for CWV: extract `LARGEST_CONTENTFUL_PAINT_MS.percentile`, `CUMULATIVE_LAYOUT_SHIFT_SCORE.percentile`, `INTERACTION_TO_NEXT_PAINT.percentile`. If `loadingExperience.overall_category` is `"INSUFFICIENT_DATA"`, set `origin_fallback: true` and try `originLoadingExperience` instead. If both are absent, `cwv: null`.
  - Parse Lighthouse: `lighthouseResult.categories.performance.score`, `lighthouseResult.categories.accessibility.score`, `lighthouseResult.categories.seo.score`, and `lighthouseResult.categories['best-practices'].score` (bracket notation required — the key literal is `'best-practices'`).
  - Return `ExternalSignals`. Wrap entire body in try/catch — return `null` on any parse error.

- [ ] **Add startup warning in `crawler/src/index.ts`** — after the `GROQ_API_KEY` hard-fail block, add:
  ```
  if (!process.env.PAGESPEED_API_KEY) {
    console.warn('[feeltrace-crawler] PAGESPEED_API_KEY not set — PSI/CWV/Lighthouse signals disabled')
  }
  ```
  This is a `console.warn` only — no `process.exit`.

- [ ] **Create `crawler/src/pipeline/stage1-external-scorer.ts`** — two exported functions:
  - `scoreExternalSignals(signals: ExternalSignals): ScoredIssue[]` — checks `signals.cwv` then `signals.lighthouse`; returns `[]` if both are null:
    - CWV rules (signal_source prefix `cwv.`, category `perceived-perf`, viewport `'both'`):
      - `lcp_ms > 4000` → severity 4; `lcp_ms > 2500` → severity 3
      - `cls_raw > 25` → severity 4; `cls_raw > 10` → severity 3
      - `inp_ms > 500` → severity 4; `inp_ms > 200` → severity 3
      - Skip a metric if its value is `null`.
    - Lighthouse rules (signal_source prefix `lighthouse.`, category per rule, viewport `'both'`):
      - `performance < 0.5` → category `perceived-perf`, severity 4; `< 0.7` → severity 3
      - `accessibility < 0.8` → category `accessibility`, severity 3; `< 0.9` → severity 2
      - `seo < 0.7` → category `technical-perf`, severity 2
    - For `raw_evidence`, format as: `"<value> (threshold: <op><threshold> <label>)"` matching the pattern in `stage1-scorer.ts`.
    - Apply highest-severity-wins deduplication per signal_source (avoid emitting both severity 4 and 3 for the same LCP value — only the highest triggered rule fires).
  - `scoreAxeViolations(violations: AxeViolation[]): ScoredIssue[]` — implemented in Plan 06-02 but the function signature must be exported here as a stub returning `[]` so Plan 06-02 can fill it in without changing the import path.

  - Import `fetchPSISignals` from `./lib/psi`.
  - Replace the current `Promise.race([runDualViewportCrawl(...), timeout])` block with two-step destructuring:
    ```
    const [crawlResult, psiResult] = await Promise.race([
      Promise.all([
        runDualViewportCrawl(url, jobId),
        fetchPSISignals(url, 30_000),
      ]),
      new Promise<never>((_, reject) =>
        setTimeout(() => reject(new Error('SLA exceeded after 55s')), SLA_MS)
      ),
    ])
    const { mobile, desktop, screenshot, techProfile } = crawlResult
    ```
  - `Promise.race` resolves to whichever branch wins. The `Promise.all` branch returns a two-element array `[crawlResult, psiResult]` — destructure at that level, then unpack `crawlResult` separately. Do NOT attempt `const [{ mobile, ... }, psiResult] = await Promise.race(...)` in one step; TypeScript cannot infer the nested tuple type through the race overload.
  - Pass `psiResult` to `runAIPipeline` as the new fifth argument (after `techProfile`).
  - NOTE: RESEARCH.md draft placed `externalSignals` as the third parameter in `runAIPipeline`; **the plans use fifth position (after `techProfile`) — fifth is authoritative. Disregard RESEARCH.md parameter ordering.**

- [ ] **Update `crawler/src/pipeline/run-pipeline.ts`** — extend `runAIPipeline` to accept and use `ExternalSignals | null`:

## Notes

- **CLS raw integer caveat:** The PSI API returns CLS as an integer percentile (e.g., `8` means CLS 0.08). The threshold rules use the raw integer: `> 25` = Critical (0.25 float), `> 10` = High (0.10 float). Do not divide by 100.
- **`best-practices` bracket notation:** The Lighthouse response key is literally `'best-practices'` with a hyphen. Use `data.lighthouseResult.categories['best-practices'].score` — dot notation will cause a TypeScript parse error.
- **Highest-severity-wins dedup:** For each signal_source (e.g., `cwv.lcp_ms`), fire only the highest rule that triggers. Iterate rules from severity 4 down to 1 and `break` after the first hit, or filter and take the max. Both approaches are acceptable.
- **PSI strategy:** Use `MOBILE` strategy for CWV to match the mobile-first crawl philosophy already established in `processor.ts`.
- **No Prisma schema change:** `_signals` is stored inside the existing `Result.tech_stack: Json` column. No migration needed.
- **`scoreAxeViolations` stub:** Export it from `stage1-external-scorer.ts` in this plan as `function scoreAxeViolations(_violations: AxeViolation[]): ScoredIssue[] { return [] }`. Plan 06-02 replaces the body without changing the export contract.
- **`signal_source` prefix — `axe.` is authoritative:** The plans use `axe.<violation.id>` as the signal_source prefix for axe violations (e.g., `axe.image-alt`). The RESEARCH.md draft used `axeViolations.<id>`. The plans' `axe.` prefix supersedes the RESEARCH.md draft — use `axe.` throughout.
- **`externalSignals` parameter position — fifth is authoritative:** Plans 06-01 and 06-02 add `externalSignals` as the fifth parameter to `runAIPipeline` (after `techProfile`). RESEARCH.md draft showed it as the third parameter. The plan is authoritative — always use fifth position.
- **SLA math:** `fetchPSISignals` is called with `30_000` ms (30s timeout). The outer `Promise.race` SLA is `55_000` ms. PSI has 25s of headroom before the outer SLA fires.
